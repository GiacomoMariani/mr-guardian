# MR Guardian Status

This file summarizes the current implemented state of the project. Historical
ticket files remain in `tickets/`, but this document should describe the
behavior that exists today.

## Implemented

### Review Pipeline

- Local review runs through the unified `review` command.
- The review command loads all YAML policy files from the configured policy
  directory.
- `--policy-dir` can override the directory, but users do not pass a single
  policy file for normal use.
- `--no-store` prints the report without writing review history.
- Deprecated `inspect` and `inspect-all` commands are not part of the active CLI.

### Runtime Policy Model

- YAML is the only runtime source of truth for rules.
- Valid top-level policy fields are `version` and `rules`.
- Every item under `rules` is executable policy configuration.
- Rules can be `type: deterministic` or `type: llm`.
- Deterministic rules map to Python implementations through the rule registry.
- LLM rules are prompt-configured, advisory, and cannot be blocking.
- Rule-specific configuration lives under `parameters`.
- Markdown guidance files are optional human reference material and are not
  loaded or validated by runtime code.

### Deterministic Rules

The project includes deterministic rules for:

- required MR metadata sections
- ticket-key enforcement for GitLab MR titles
- changed file and changed line thresholds
- broad directory changes
- Unity scene, prefab, ProjectSettings, and gameplay validation requirements
- C# debug logs
- C# `GetComponent`
- C# public fields
- C# class, method, and parameter size checks
- Python `print` calls
- Unity event lifecycle risks
- Unity per-frame allocation patterns
- Unity pooling concerns
- Unity `Resources.Load`

### LLM Rules

LLM rules are loaded from YAML and executed when OpenAI configuration is
available. They use structured output, retry/timeout settings, defensive
severity normalization, and per-rule usage metrics when the provider reports
them.

When dependencies, provider configuration, rate limits, malformed output, or API
errors prevent execution, MR Guardian records advisory `info` findings and
continues deterministic review.

### Reporting

Reports include:

- overall risk
- developer identity when available
- coding and MR-structure evaluation summaries
- severity counts
- changed file and line counts
- reviewer focus
- finding overview
- loaded policy summaries
- grouped findings with rule IDs, sources, rule type, and locations
- LLM runtime and token metrics when available

Noise-control limits keep repeated non-blocking findings readable.

### Storage

SQLite review history stores:

- review scope
- timestamp
- project, branch, MR ID, and commit SHA when available
- developer ID
- ticket key extracted from MR title
- policy version
- overall risk and severity counts
- coding and MR-structure evaluation data
- changed file and line counts
- triggered rule IDs
- generated report
- review score
- LLM usage metrics

Manual JSON review submissions can be validated and stored through
`submit-manual-review`.

### GitLab Integration

The FastAPI service exposes `POST /webhooks/gitlab`.

It validates GitLab webhook secrets when configured, accepts opened or reopened
Merge Request events, prepares a local worktree, runs the same review pipeline,
stores the result, and can post the generated report back to GitLab when comment
posting is enabled.

GitLab automation still depends on a local or mounted Git checkout for fetch and
diff collection.

### Dashboard

The Streamlit dashboard reads SQLite review history and exposes:

- recent review history
- risk trends
- most triggered rules
- PM delivery status based on ticket-linked reviews
- recurring blocker highlights
- lead-focused developer performance and review-attempt trends
