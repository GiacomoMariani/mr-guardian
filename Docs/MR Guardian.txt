# MR Guardian — Unity MR Review Assistant

## Goal

Build a portfolio-ready tool that reviews Unity Merge Requests against team-specific engineering best practices.

**Positioning:**

> MR Guardian is an AI-assisted workflow tool that helps Unity teams catch risky, incomplete, or poorly documented Merge Requests before human review.

The tool does **not** replace human review.

It acts as a workflow assistant that checks whether an MR is ready for review, highlights risky Unity-specific changes, stores review history, and points developers back to the team's best-practices documentation.

## 3. Policy Architecture

MR Guardian uses both Markdown and YAML.

```text
Markdown = human-readable best-practices source
YAML     = machine-readable policy
Rule IDs = stable link between both
```

Markdown explains the rule:

```markdown
## UNITY-SCENE-001 — Scene changes require manual validation

Unity scene files can introduce runtime behavior changes that are hard to review from code alone.

Every MR that changes or adds a `.unity` scene must include manual validation notes.
```

YAML makes the rule executable:

```yaml
- id: UNITY-SCENE-001
  enabled: true
  severity: high
  source: UnityBestPractices.md#UNITY-SCENE-001
  description: Scene changes require manual validation.
  match:
    changed_files:
      - "Assets/**/*.unity"
  require:
    mr_sections:
      - Manual Validation
  suggested_fix_ref: manual_validation
```

Every finding should reference the rule ID:

```text
⚠️ [UNITY-SCENE-001] Scene changed without manual validation.
Source: UnityBestPractices.md#UNITY-SCENE-001
```

## 8. Rule Packs to Implement

## 8.1 MR Readiness Rules

Check that the MR includes:

- title
- summary
- linked issue, card, or bug
- test plan
- risk notes
- reviewer notes
- rollback notes when applicable
- video or screenshot evidence for visible Unity changes

When required metadata is missing, suggest a fix.

Example suggested section:

```markdown
### Test Plan

- [ ] Confirm the project compiles.
- [ ] Run the affected Unity scene.
- [ ] Test the main success path.
- [ ] Test at least one edge case.
- [ ] Check the Unity Console for errors.
- [ ] Attach a video or screenshot if the change is visual.
```

---

## 8.2 Unity Asset Risk Rules

High-priority paths:

```text
Assets/**/*.unity
ProjectSettings/**
Packages/manifest.json
Packages/packages-lock.json
Assets/Plugins/**
Assets/ThirdParty/**
```

Medium-priority paths:

```text
Assets/**/*.prefab
Assets/**/*.asset
Assets/**/*.asmdef
Assets/Resources/**
Assets/Addressables/**
```

Rules:

- scene changes are high importance
- added or deleted scenes are high importance
- prefab changes are warnings unless they touch core prefabs
- core prefab changes are high importance, the core prefabs will be shown in yaml
- ProjectSettings changes require explicit explanation
- package changes require risk notes
- third-party edits should be flagged


---

## 8.3 Unity Test and Playtest Rules

Production paths:

```text
Assets/Scripts/**
Assets/**/Scripts/**
```

Test paths:

```text
Assets/Tests/**
Assets/**/Tests/**
Tests/**
```

Rules:

- gameplay code changed but no test files changed
- scene changed without manual validation
- prefab changed without manual validation
- visual change missing video or screenshot
- bug fix missing root-cause explanation

Example suggested section:

```markdown
### Manual Validation

- [ ] Opened the affected scene.
- [ ] Entered Play Mode.
- [ ] Verified the changed behavior.
- [ ] Checked the Unity Console for errors.
- [ ] Attached a video or screenshot if visual behavior changed.
```

---

## 8.4 C# Best-Practice Rules

Initial checks:

- class too long
- method too long
- method has too many parameters
- public fields in MonoBehaviours
- `Debug.Log` added
- `print` added
- `GetComponent` used outside `Awake`
- `GetInstanceID` used
- direct edits inside third-party folders
- long method chains
- ScriptableObject appears to hold runtime mutable state

Example finding:

```text
⚠️ [CSHARP-GETCOMPONENT-001] PlayerHealthView.cs uses GetComponent outside Awake.
```

---

## 8.6 MR Size and Focus Rules

Warn when an MR is too large or unfocused.

Checks:

- too many changed files
- too many changed lines
- too many unrelated directories
- scene or prefab changes mixed with broad code refactors
- generated files mixed with hand-written logic
- gameplay, art, tooling, and settings mixed without explanation

---

## 8.7 AI-Assisted Code Risk Rules

Detect patterns that often happen when AI-generated code goes off track.

Checks:

- MR is much larger than the ticket scope
- unrelated files changed
- generated helper abstractions with no clear use
- duplicated logic
- inconsistent naming
- excessive public APIs
- obvious comments explaining simple code
- temporary debug logs
- code added without tests or manual validation

Suggested section:

```markdown
### AI-Assisted Development Notes

- [ ] Confirmed all changed files are required.
- [ ] Removed unused generated code.
- [ ] Removed temporary debug logs.
- [ ] Checked the implementation against team best practices.
- [ ] Added tests or manual validation notes.
```

## 11. Minimal Policy Example

```yaml
version: 1

best_practices:
  source_url: "https://github.com/GiacomoMariani/UnityBestPractices"
  local_markdown_path: "best-practices/UnityBestPractices.md"
  require_rule_id_links: true

history:
  enabled: true
  database_path: ".mr-guardian/history.sqlite"

mr_metadata:
  required_sections:
    - Summary
    - Linked Issue
    - Test Plan
    - Risk
    - Reviewer Notes

  suggested_sections:
    test_plan: |
      ### Test Plan

      - [ ] Confirm the project compiles.
      - [ ] Run the affected Unity scene.
      - [ ] Test the main success path.
      - [ ] Check the Unity Console for errors.

    manual_validation: |
      ### Manual Validation

      - [ ] Opened the affected scene.
      - [ ] Entered Play Mode.
      - [ ] Verified the changed behavior.
      - [ ] Checked the Unity Console for errors.

size:
  max_changed_files_warning: 15
  max_changed_lines_warning: 500
  max_directories_warning: 5

unity:
  scene_paths:
    - "Assets/**/*.unity"
  prefab_paths:
    - "Assets/**/*.prefab"
  project_settings_paths:
    - "ProjectSettings/**"
  package_paths:
    - "Packages/manifest.json"
    - "Packages/packages-lock.json"
  production_code_paths:
    - "Assets/Scripts/**"
    - "Assets/**/Scripts/**"
  test_paths:
    - "Assets/Tests/**"
    - "Assets/**/Tests/**"
    - "Tests/**"

csharp:
  max_class_lines: 300
  max_method_lines: 50
  max_method_parameters: 5
  flag_public_fields: true
  flag_debug_log: true
  flag_get_component_outside_awake: true

rules:
  - id: MR-META-001
    enabled: true
    severity: blocking
    source: UnityBestPractices.md#MR-META-001
    description: MR must include a test plan.
    require:
      mr_sections:
        - Test Plan
    suggested_fix_ref: test_plan

  - id: UNITY-SCENE-001
    enabled: true
    severity: high
    source: UnityBestPractices.md#UNITY-SCENE-001
    description: Scene changes require manual validation.
    match:
      changed_files:
        - "Assets/**/*.unity"
    require:
      mr_sections:
        - Manual Validation
    suggested_fix_ref: manual_validation

  - id: UNITY-PREFAB-001
    enabled: true
    severity: warning
    source: UnityBestPractices.md#UNITY-PREFAB-001
    description: Prefab changes require validation evidence.
    match:
      changed_files:
        - "Assets/**/*.prefab"

  - id: UNITY-PROJECTSETTINGS-001
    enabled: true
    severity: high
    source: UnityBestPractices.md#UNITY-PROJECTSETTINGS-001
    description: ProjectSettings changes require explanation.
    match:
      changed_files:
        - "ProjectSettings/**"
    require:
      mr_sections:
        - Risk
        - Reviewer Notes

  - id: UNITY-TESTS-001
    enabled: true
    severity: warning
    source: UnityBestPractices.md#UNITY-TESTS-001
    description: Gameplay code changes require tests or manual validation.
    match:
      changed_files:
        - "Assets/Scripts/**"
        - "Assets/**/Scripts/**"
    require_any:
      changed_files:
        - "Assets/Tests/**"
        - "Assets/**/Tests/**"
        - "Tests/**"
      mr_sections:
        - Manual Validation

  - id: CSHARP-DEBUG-001
    enabled: true
    severity: warning
    source: UnityBestPractices.md#CSHARP-DEBUG-001
    description: Avoid committed Debug.Log or print statements.
    match:
      changed_files:
        - "Assets/**/*.cs"
      added_lines_contain:
        - "Debug.Log"
        - "print("

  - id: AI-CODE-001
    enabled: true
    severity: warning
    source: UnityBestPractices.md#AI-CODE-001
    description: AI-assisted code should not add unnecessary scope.
    match:
      changed_files_count_greater_than: 15
```

# Part 2 — Why This Is the Right Portfolio Piece

This section explains the reasoning.

---

## Why This Matters

Unity code review is difficult because important changes are not always obvious from code alone.

A Unity MR can touch scenes, prefabs, serialized assets, ProjectSettings, packages, inspector references, generated metadata, gameplay scripts, and editor scripts.

MR Guardian helps reviewers focus on risky areas before deep code review.

---

## Why This Matters for AI-Assisted Programming

AI coding tools can generate useful code quickly, but they can also create changes that are too broad, under-tested, inconsistent, or difficult to review.

This is especially risky in Unity projects because AI-generated changes may touch scenes, prefabs, serialized assets, ProjectSettings, or inspector references.

Junior programmers may miss these issues because generated code can look plausible while still violating project standards.

MR Guardian helps by:

- highlighting unfocused MRs
- detecting risky Unity files
- flagging missing tests or validation evidence
- warning about shortcuts like public fields, `Debug.Log`, or uncached `GetComponent`
- showing junior developers why a change is not review-ready
- giving reviewers a focused summary of where AI-generated work may have gone off track

The goal is not to block AI-assisted programming.

The goal is to make it safer, more reviewable, and more aligned with team standards.

---

## Why Store Review History?

Single MR reports are useful.

Review history is more valuable.

By storing review history, MR Guardian can answer questions like:

- Are MRs getting cleaner over time?
- Which rules trigger most often?
- Are missing test plans common?
- Are scene or prefab changes frequently under-documented?
- Are AI-generated MRs larger or riskier?
- Which areas need team training?

This turns MR Guardian from a one-off reviewer into a workflow improvement tool.

---

## Why Markdown and YAML?

Markdown is for humans.

YAML is for the tool.

Rule IDs connect both.

```text
Markdown explains the rule.
YAML enforces the rule.
Rule IDs make findings traceable.
```

This keeps the system maintainable, explainable, configurable, and testable.

---

## Why Deterministic Checks First?

AI-only review can be inconsistent.

MR Guardian should use deterministic rules for findings and AI only for summarization.

```text
Rules find issues.
AI explains and prioritizes them.
Humans make the final decision.
```

## FAQ

## If I already have a Markdown best-practices document, how do I convert it to YAML?

Do not convert the whole document.

Extract only the parts that can become executable rules.

Process:

1. Add stable rule IDs to Markdown headings.
2. Identify which rules can be checked automatically.
3. Create one YAML rule per checkable rule.
4. Keep explanations and examples in Markdown.
5. Keep patterns, thresholds, severity, and suggested fixes in YAML.

## Should YAML replace Markdown?

No.

Markdown explains. YAML enforces.

## Can AI help convert Markdown into YAML?

Yes, but the result should be reviewed manually.

Check that:

- rule IDs are stable
- severity is correct
- file patterns are correct
- the rule is enforceable
- suggested fixes are useful

## What if a Markdown rule has no YAML rule?

That is fine.

Some best practices are reviewer guidance only.

## What if a YAML rule has no Markdown source?

MR Guardian should warn or fail when this is enabled:

```yaml
best_practices:
  require_rule_id_links: true
```

## Why not only use AI to review the MR?

Because AI-only review can be inconsistent.

MR Guardian uses rules for reliability and AI for explanation.

## How does this help junior developers?

It teaches them:

- what rule was triggered
- why it matters
- where it is documented
- what to fix before review

## How does this help project managers?

It shows trends over time:

- repeated blockers
- risky areas
- missing validation patterns
- review readiness
- AI-generated code risk

## How does this help with AI-generated code?

It catches common AI-code problems:

- excessive scope
- missing tests
- risky Unity asset changes
- temporary logs
- inconsistent naming
- missing validation evidence
- changes that do not match the MR description
